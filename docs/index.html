01 — Vue d'ensemble

Rôle de l'API

L'API CampusRoom est le cœur backend du système de gestion des campus, des salles et des réservations. Elle expose des endpoints RESTful qui permettent de :

  • Gérer l'infrastructure physique : création et consultation de campus et de salles.
  • Gérer les utilisateurs (étudiants, enseignants) avec leurs rôles et quotas.
  • Orchestrer les réservations : le domaine le plus complexe. L'API gère différents types (Cours, Réunion, Examen) et applique des règles métier strictes — conflits d'horaires, quotas, priorités — avant toute validation.

02 — Structure

Architecture Multicouche

L'application est structurée en couches bien séparées, chacune avec une responsabilité unique :

Controllers
Interceptent les requêtes HTTP et délèguent au service — ReservationController, RoomController, CampusController, UserController
Services
Contiennent toute la logique métier. Orchestrent Factory et Strategies. Interfaces + implémentations séparées.
Design Patterns
ReservationFactory + ValidationStrategy — gèrent la création et la validation des réservations de façon modulaire
Recherche / Spec
JPA Specifications + pagination — ReservationSpecification, RoomSpecification, CampusSpecification, UserSpecification
Repositories
Interfaces JPA (JpaRepository + JpaSpecificationExecutor) pour la persistance en base H2
Entités JPA
Campus → Room → Reservation ← User — relations OneToMany / ManyToOne avec cascade

04 — Design Patterns

Factory Pattern & Strategy Pattern

Pour traiter la complexité des règles de création et de validation des réservations, l'architecture s'appuie sur deux patterns complémentaires.

Factory Pattern

ReservationFactory
CourseReservation
MeetingReservation
ExamReservation
«interface» ReservationBehaviorgetDescription() · getMaxDurationHours()

Strategy Pattern

«interface» ValidationStrategyvalidate(reservation, user)
ConflictStrategyconflits horaires
PriorityStrategyprof > étudiant
QuotaStrategylimite mensuelle
ReservationServiceitère sur toutes les strategies injectées

Factory — pourquoi ?

Sans Factory, chaque nouveau type de réservation impliquait un if/else supplémentaire dans le service. La ReservationFactory centralise l'instanciation : le service demande un objet pour le type donné, sans connaître la classe concrète. Chaque implémentation porte ses propres règles (getMaxDurationHours(), description…).

Strategy — pourquoi ?

Chaque règle de validation est une classe @Component indépendante. Spring les injecte automatiquement dans une List<ValidationStrategy>. Le service itère sur la liste — ajouter une nouvelle règle se résume à créer une classe, sans toucher au service existant. C'est l'Open/Closed Principle appliqué.
